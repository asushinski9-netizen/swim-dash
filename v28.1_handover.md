# v28.1 Handover — QT Schema Migration, Error Handling & Inline QT Editor

## Scope of this release
This release covers three layered changes on top of the v27.1 baseline:

1. **v28** — JSON error handling, `se_london_qt.json` → `regional_qt.json` rename, QT schema migration to `{meta, times}`.
2. **v28.1 (first pass)** — Moved the QT editor from standalone tabs into an inline edit mode inside the County QT / Regional QT tabs themselves, plus six UX fixes (time format, duplicate detection, not-offered greying, full stroke names, mobile banner, back-button flow).
3. **v28.1 (this pass)** — Two small banner-copy consistency fixes.

The current file is **`index.html`**, single-file, ~4512 lines.

---

## 1. JSON error handling (v28)

- Added `unwrapQT(raw)` — accepts either the old flat-array QT schema or the new `{meta, times}` schema and always returns `{meta, times}` internally.
- Added `showJsonLoadError(filename, message)` — renders a persistent red banner at the top of the Overview tab when a JSON file fails to parse or fetch.
- Wrapped every `JSON.parse()` call (RAW, QT_DATA, SE_QT_DATA, UPCOMING — both from `fetch()` and from `localStorage`) in try/catch.
- `saveDataToBrowser()` / its internal `readFile()` now show a rich in-modal error instead of a bare `alert()` when an uploaded file fails to parse.

## 2. Regional file rename (v28)

`se_london_qt.json` → `regional_qt.json` everywhere: the `SE_QT_DATA_URL` constant, Data Manager upload label, empty-state copy, and bracket notes. (Internal JS variable names `SE_QT_DATA` / `SE_QT_META` were kept as-is to minimise risk — only the **file name** and **user-facing label** changed.)

## 3. QT schema migration (v28)

New schema per `qt_schema.md`:

```json
{
  "meta": { "title": "...", "dateFrom": "YYYY-MM-DD", "dateTo": "YYYY-MM-DD" },
  "times": [ { "gender": "...", "course": "...", "event": "...", "age": "...", "qualify": 38.10, "consider": 40.80 }, ... ]
}
```

- New globals `QT_META` and `SE_QT_META`, populated from `.meta` on load.
- `QT_DATA` / `SE_QT_DATA` remain **flat arrays** (set to `.times`), so every existing render function (`renderQualifying`, `renderRegionalQualifying`, `getQTStatusForEvent`, `renderTargets`, etc.) is untouched.
- Old flat-array files still load fine via `unwrapQT()` — backward compatible.
- Bracket notes (`#qtNote` / `#rqNote`) now show `meta.title` instead of a hardcoded championship name string.

## 4. Inline QT Editor — replaces standalone editor tabs (v28.1)

### What changed structurally
The two original standalone tabs (`County QT Editor`, `Regional QT Editor`) were **removed entirely**. Editing now happens inline, inside the County QT and Regional QT tabs themselves.

Each of `#tab-qualifying` and `#tab-regional` is now split into two mutually-exclusive sections:

```
#tab-qualifying
 ├─ .qt-meta-banner          (always visible — title, dates, Edit button)
 ├─ #qtViewMode               (existing filters/cards/table — shown by default)
 └─ #qtEditMode                (inline editor — shown only in edit mode)
```

Mode switching is done by toggling a `.qt-tab-in-edit-mode` class on the parent `.panel` div:
- `openQTEditor('qt' | 'rq')` — adds the class, deep-clones the live `{meta, times}` data into an in-memory `qtEditorStore` (only on first open per session), populates the Age dropdown, renders the editor table.
- `closeQTEditor('qt' | 'rq')` — removes the class, commits the editor store back into the live `QT_DATA`/`SE_QT_DATA`/`QT_META`/`SE_QT_META` globals via `commitEditorStoreToLive()`, refreshes the banner, and re-renders the view tab fresh (no page reload needed).

Editor element IDs use a **separate prefix** (`cqt`/`rqt`) from the view-tab prefix (`qt`/`rq`) so there's never an ID collision. `editorPrefixFor(viewPrefix)` maps between them.

### Meta banner
A `.qt-meta-banner` sits above the filters in both tabs, always visible regardless of screen size:
- Shows the championship title and date range (or a "no dates set" note — see §6 below).
- Shows an `✏️ Edit Details & Times` button — **desktop only** (`@media (min-width: 769px)`), since the editor table isn't designed for small screens.
- `renderQTBanner('qt' | 'rq')` populates it; called from `showTab()`, `init()`, and after `saveMetaPanel()`.

### Time format (was raw seconds)
Qualify/Consider columns in the editor now accept `mm:ss.hh` or `ss.hh` (e.g. `2:44.68`, `41.88`), matching the format used throughout the rest of the dashboard:
- `secToTimeInput()` wraps the existing `secToTime()` helper.
- `isValidTimeInput()` — regex `^(\d{1,3}:)?\d{1,3}(\.\d{1,2})?$`; empty string is valid (= not offered).
- `timeInputToSec()` wraps the existing `timeToSec()` helper.
- Invalid input gets a red border + tooltip; conversion is round-trip safe (verified: 38.10↔"38.10", 164.76↔"2:44.76", 1225.05↔"20:25.05", etc.).

### Duplicate detection
`findDuplicateIndices(times)` checks the **entire dataset** (not just the currently filtered view) for rows sharing the same Gender + Course + Event + Age. Matches get:
- An amber `.has-duplicate` row highlight.
- A `⚠ Duplicate` badge next to the Event dropdown.
- A confirmation warning before `downloadEditorJSON()` proceeds if any duplicates remain.

### Not-offered rows
Rows where **both** Qualify and Consider are null/undefined are visually de-emphasised:
- `.not-offered` CSS class (~45% opacity).
- All inputs/selects and the delete button get `disabled`.
- A note: *"Not offered at this age — leave blank, or enter a time to make it available."*
- Entering a time in either field re-enables the row on next render.

### Full stroke names
Editor Stroke filter dropdowns now read Freestyle / Backstroke / Breaststroke / Butterfly / Individual Medley — matching the view-tab filter labels exactly (previously abbreviated Free/Back/Breast/Fly/IM).

### Championship details panel
`toggleMetaPanel(prefix)` shows/hides a small form (Title, Date From, Date To) inside the editor. `saveMetaPanel(prefix)` writes into the in-memory editor store **and** immediately commits to the live globals + refreshes the banner — so title/date edits are reflected instantly even before clicking "Back".

### Download JSON
`downloadEditorJSON(prefix)` always reflects the current in-memory editor store (including any edits not yet committed via "Back"). Warns before downloading if rows are missing Event/Age, or if duplicates are detected. Produces a file matching the `{meta, times}` schema, named `county_qt.json` or `regional_qt.json`.

### ⚠️ Known caveat (not yet addressed)
The editor store (`qtEditorStore.cqt` / `qtEditorStore.rqt`) is only committed to the live `QT_DATA`/`SE_QT_DATA` globals when **"Back"** is clicked or when the meta panel's **"Save"** is clicked. If a user navigates away from the QT tab (e.g. clicks a different top-level tab) while edits are unsaved and never clicks "Back", those row-level edits will be lost on next `openQTEditor()` call (since the store is only initialised once per session, but is never re-synced from a different commit point). This has **not yet been flagged to the user** in-app — worth considering for a future pass (e.g. a "Save & Continue Editing" button, or saving on every keystroke).

## 5. `showTab()` / `init()` updates (v28.1)

- `showTab()` no longer has `county-editor` / `regional-editor` cases (the standalone tabs are gone). The `qualifying` and `regional` cases now call `renderQTBanner('qt')` / `renderQTBanner('rq')` before the existing render call.
- `init()` now calls `renderQTBanner('qt')` and `renderQTBanner('rq')` immediately after data load (before `renderQualifying()`/`renderRegionalQualifying()`), so the banner is populated on first page load even though Overview is the default active tab.

## 6. Banner copy consistency (v28.1, this pass)

Two small copy fixes to the "no dates set" message inside `.qt-meta-banner`:

**Before:**
> No dates set — click Edit Details to configure

This was shown on **all** screen sizes, but:
- The actual button is labelled **"Edit Details & Times"**, not "Edit Details" — inconsistent.
- On mobile, there is no Edit button at all (desktop-only), so the instruction was actionable only on desktop, yet the same text showed everywhere.

**After**, two variants are rendered into the DOM and CSS shows the correct one per breakpoint (same `769px` threshold as `.btn-edit-qt`):

| Breakpoint | Message |
|---|---|
| Desktop (≥769px) | *No dates set — click "Edit Details & Times" to configure* |
| Mobile (<769px) | *No dates set — switch to Desktop mode to edit* |

Implementation: `renderQTBanner()` now sets `datesEl.innerHTML` (not `.textContent`) with two `<span>`s — `.qt-banner-note-desktop` and `.qt-banner-note-mobile` — toggled via a small new CSS block reusing the existing `min-width: 769px` breakpoint already used for `.btn-edit-qt`. No JS viewport detection needed; pure CSS show/hide, consistent with how `.col-full`/`.col-abbr` and `.comp-full`/`.comp-short` already work elsewhere in the codebase at the 500px breakpoint.

This only affects the **no-dates** message. Once a title/dates are set, the banner shows the formatted date range as before and the note never appears.
